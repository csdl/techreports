Inca Software Requirements Specification

Architectural Issues

Three Server-side Tiers  

The first, fundamental architectural property of Inca is that it is designed with three server-side tiers. This means that server-side services and functions can be distributed across three distinct computers communicating with each other across a network, as illustrated in the following figure:

The Web Tier contains one or more computers running a web server with an embedded Java process that can handle servlets and JSP pages. Examples include Apache, TomCat, or Resin. The Web Tier serves pages to the client by combining together formatting information in the JSP page with dynamic content provided to the page in the form of JavaBeans (here called JSP Beans, to indicate that they are JavaBeans specifically for use in JSP pages). The Web Tier also contains  Java classes that implement the web tier components of the "controller" in the MVC design pattern. 

The Web Tier MVC controller has two basic tasks. First, the controller must process each user request, and if the request requires processing not possible at the Web Tier,  then generating an "event" that can be sent to the EJB tier for further manipulation. The results of such processing are communicated by the EJB Tier back to the Web Tier through "Info objects", which update the internal state of the JSP Beans.  The second task of the Web Tier MVC controller is to decide which JSP page should be presented to the user next, based upon the last page presented and the current state of the JSP Beans after updating via the info objects.

The EJB (Enterprise JavaBean) Tier contains an EJB container with stateless session, stateful session, and entity beans. These beans implement the business logic for the application and the interactions with the persistent storage in the EIS Tier. One of the session beans has a special purpose: it provides the EJB Tier components of the MVC controller, and is responsible for receiving the event generated from the Web Tier’s MVC controller, invoking other EJBs as necessary to accomplish the task specified by the event, and communicating the results back to the Web Tier so that the appropriate info objects can be sent to update the JSP Beans. When the actions of the Enterprise JavaBeans result in changes to the persistent store, then SQL queries are generated to communicate with the EIS Tier from the EJB Tier.  

The Enterprise Information System (EIS) Tier contains a relational database system such as Cloudscape, Oracle 8i, or mySQL. This system contains a set of tables that implement the persistent storage for the information in the Inca system. The EIS tier accepts SQL queries sent from the EJB Tier, and returns the SQL results. 

Two final comments may help avoid confusion. First, although Inca is designed as a server-side three tier system, it does not require each tier to reside on physically distinct computers. For example, the J2EE reference implementation provides an "all-in-one" solution, where the (Web Tier) web server, the (EJB Tier) EJB container, and the (EIS Tier) relational database system can all be run simultaneously on one computer. However, from an architectural standpoint, the software on each tier runs in its own process and requires network access to communicate with the other tiers. 

Second, the enterprise computing literature frequently refers to "three tier" systems. However, in this literature the three tiers are actually referring to a "thin client tier" (i.e. a web browser), which communicates via a network to the "middle tier" (some sort of middleware that implements the business logic), which communicates via a network to the "database tier" (that contains one or more relational database servers). In the description of the Inca architecture above, the "client tier" containing the web browser is omitted for clarity’s sake, although it is certainly part of the overall application structure. Thus, from the standpoint of the enterprise computing literature, the Inca architecture is a "four tier" system, which would normally be referred to in the literature as "n-tier". When speaking to "outsiders" about the Inca system, therefore, it is most descriptive to refer to the Inca as an n-tier system. However, from the developer's standpoint, we only need concern ourselves with the three, server-side tiers.

The user interaction cycle in detail

Now that the overall architectural structure of Inca is understood, it will be useful to go into a bit more detail about the way in which information flows through the tiers. The following diagram presents our three server-side tiers again, this time with more detail about their internal processing characteristics.

Let's step through the sequence of events that occur when a user interacts with the system:

  1. A JSP page is served to the user’s browser and displayed. The user fills out a form or clicks on a link, resulting in an HTTP request being generated and sent to the web tier’s web server. This request is handled by a single servlet instance, which takes the request object containing information about the user’s interaction and gives it to the RequestProcessor.
      
  2. The RequestProcessor is a session-based property, and thus each user has their own instance that can store user-specific state. It takes the request object from the servlet, extracts the relevant information, and constructs an event instance. The event instance provides a more convenient representation for the intended user effect, and is also serializable so that it can be sent over the network from the web tier to the EJB tier. The RequestProcessor hands off the event to the Web Tier controller.
      
  3. The Web Tier controller receives the event from the request processor through its handleEvent method. It then invokes the handleEvent method of the EJB Tier controller. This results in the event being transmitted over the network to the EJB tier. Once there, the EJB controller inspects the event to determine what should happen, and invokes the other EJB instances as needed. For example, if the event is a login event, then other EJB instances are invoked to validate the user’s account name and password, which may require accessing the EIS tables. In any case, the results of whatever processing occurs are ultimately reflected in changes to one or more of the Model objects in the EJB layer. The EJB Tier controller creates a list of the changed models and sends that back as the return value to the Web Tier controller.
      
  4. Once the Web Tier knows which EJB Tier models have changed, the next task is to update the JSP Bean objects. There is a one-to-one correspondence between Web Tier JSP Bean instances and EJB Tier model instances. So, the Web Tier controller takes the list of updated models it received and passes it off to the ModelUpdateNotifier. This notifier invokes the performUpdate method of the JSP Beans which implements calls to the EJB Tier to synchronize their state. They do this by receiving one or more"Info" instances from the Models, which are serializable value objects containing the new data.
      
  5. Once the Web Tier JSP Beans are resynchronized with the EJB Tier models, it is time to select the next JSP page to present to the user. The Web Tier controller invokes the ScreenFlowManager, which determines the next page based upon the current page and current state of the JSP Beans. The page is served. Finally, the user fills out a form and presses submit, or clicks on a link, which generates the next request object in the servlet, and the cycle continues.

The following sections provide a further level of detail in the description of the system, by presenting specifications for the JSP Pages, JSP Beans, models, Info objects, Events, EJBs, and their interrelationships.

This table illustrates the following characteristics.  First, naming conventions include the following: JSP pages are capitalized in a manner similar to Java class names; JSP Bean class names end with the suffix "JSPBean"; and events are named with the suffix "Event".

Second, the request generated by the user falls into two basic categories: GoTo requests and Event requests. The first category, "GoTo", is simply a request to go to a different page without otherwise affecting the state of the application.  GoTo requests can generally be processed using state information in the Web Tier, and thus do not generate events.   This does not mean the request is automatically satisfied. For example, a request to GoToEditStudent would not be satisfied if the session does not include state information indicating that the user is an administrator who has successfully logged in. GoTo requests are represented here by the prefix "GoTo".  

Event requests require communication with the EJB Tier to retrieve data, change data, or both.  An Event request, as its name implies, results in the generation of an event object that is sent across the network from the Web Tier to the EJB Tier for processing. Most event objects contain additional state information required by the EJB Tier to process the request. Note that an Event request typically results in a new page being displayed, so in effect there's an implicit "GoTo" request within each Event request.

In summary, the Web Tier JSP Pages present information to the user, and each supports a defined set of user actions.  Some of these actions can be handled in the web tier exclusively, while others require interaction with the EJB tier.   The basic object for communication "downward" from Web Tier to EJB Tier is the Event object.

The next section takes a more detailed look at the structure of the JSP Beans in the Web Tier, which provide insight into how the state information in the system is organized for presentation to the user. The following section presents the "Info" objects, which are analogous to Event objects in that they are the basic object for communication "upward" from the EJB Tier to the Web Tier.

The Web Tier JSP Beans

The Web Tier JSP Beans must be designed to provide all the dynamic information required by the JSP pages.  The following table provides a high-level overview of the beans used by JSP pages to provide dynamic content to the user. There are a few general things to note about this table.  First, every bean used by a JSP page has the suffix "JSPBean". This is done to make the Bean API for JSP designers perfectly clear: any Java class with the suffix JSPBean is intended for use by JSP page designers, and no class without the JSPBean suffix is available for use by JSP page designers.

Second, every JSP Bean has a corresponding class in the EJB Tier that contains the same state information.  These classes always have the suffix "Model".  Thus, the ApprovalTableJSPBean in the Web Tier has a corresponding ApprovalTableModel class in the EJB Tier.  An essential part of system processing involves keeping the state of these pairs of objects synchronized.  

Third, every JSP Bean has a specific scope. Those with "application" scope provide general state information available to all users, both student and administrator. These beans are stored in the ServletContext, and they are singletons. The remaining beans have "session" scope. These beans provide state information that is specific to an individual user (who may be an administrator or a student), and so a separate instance of the bean is created for each session. These beans are stored in the HTTPSession.

The internal state information maintained by a JSP Bean is structured at the top level in one of three ways: as a table of information indexed by some sort of key, as a record structure, or as a list.  Regardless, part or all of the internal state of a JSP Bean consists of one or more "Info" objects. This is because Info objects are the "transport mechanism" by which data is sent from the EJB Tier back to the Web Tier.  Internal state in a JSP Bean that is not represented by an Info object is data that is created and maintained exclusively at the Web Tier and does not require any interaction with the EJB Tier.  

An example of information that is maintained only at the Web Tier is the "Ranking" data value.  Ranking appears in both the StudentStatusJSPBean and the QualificationTableJSPBean.  It indicates the position in the ordered list of requests occupied by that particular student's request.  Since Ranking can be easily derived from the RequestTableJSPBean (which, for each course, maintains a list of requests sorted by Priority value), it is not a part of any Info object.  

Another example is "currStudentLogins" and "currAdminLogins".  It is interesting to note that maintaining these two variables at the web server tier works only because we assume a single web server process.  If Inca grows to require multiple front-end web servers, then login state information will need to be moved into the EJB tier.

The fact that StudentStatusJSPBean is listed with two scopes might be confusing and thus warrants further explanation.  What this indicates is that each student will have their own session-scoped instance of a StudentStatusJSPBean which contains information about their status.  In addition, administrators need access to the same information in order to carry out the Edit Student page.  Thus, administrators have their own session-scoped instance which will hold an instance containining data about whatever student is being edited by the administrator.  

Info objects: Communicating between Web and EJB Tiers

As explained above, Info objects are fine-grained value objects that package together state information that must be sent (in serialized form) from the EJB tier to the Web tier. In Inca, Info objects are denoted by classes with the suffix "Info".  The internal structure of Info objects often, but not always, reflects the structure of a single row in one of the underlying relational tables in the EIS Tier. It might be helpful at this point to pop up a level and review the relationship between JSP Pages, JSPBeans, Events, Info objects, and models.  The following diagram might help make these relationships clear:

The most important points are:
Each JSP Page can use one or more JSP Bean instances.
There is a one-to-one relationship between JSP Beans in the Web Tier and Models in the EJB Tier.
A model updates its corresponding JSP Bean by sending one or more Info objects from the EJB Tier to the Web Tier.
Models are maintained by EJBs.  Some EJBs might maintain multiple models, while other EJBs might not maintain any models. However, each model is maintained by exactly one EJB.
The basic mechanism for communication from the Web Tier "down" to the EJB Tier is through an event object, which usually contains internal state.

This concludes the discussion of the architectural features of the Web Tier and its interface to the EJB Tier through events and info objects.  We now turn to the architecture of the EJB Tier.

Additional EJB Tier components: AuthenticationMgr, MessageMgr, PriorityMgr

In addition to the session and entity beans, there are additional components required to provide auxiliary services in Inca.  A top level description of their requirements follows:

Component Requirements
AuthenticationMgr Provides authentication services for new student registrations.  Uses the open source Java Telnet Application  to support connection to UHUnix to determine if user names and accounts are valid. 
MessageMgr Provides log-oriented messaging services to both the Web and EJB Tiers.  Inca requires the following kinds of logs and message services:  
Toggable console output messages to support debugging and maintenance.
File-based messages to record system activity for intrusion detection and other security activities
Email messages to students to inform them of registration success/failure and to provide them with approval codes.
Email messages to the Inca account to provide a log of registration actions in case of catastrophic failure.

Note that the same system event (i.e. a registration attempt) could result in activity at several or all of these levels.  Therefore, the MessageMgr service consolidates all output-related services within one API using the Log4J open source utility in combination with the JavaMail package.
PriorityMgr Provides the fine-grained priority values associated with a student's request for a given course, given their course background. 

From Session Beans to the EIS Tier: An Entity Bean-based solution

Session beans must communicate in some fashion with the underlying persistent store in the EIS Tier.  There are many different possible approaches, from hand-rolled JDBC, to JDO, to Entity Beans with Bean Managed Persistence (BMP), to Entity Beans with container managed persistence (CMP).  These approaches compare and contrast mainly in terms of performance vs. ease of development and enhancement.

In the initial implementation of Inca, the main goal is to validate the requirements, not performance.  Thus, from an architectural perspective, the goal is to minimize the amount of developer time and energy devoted to persistence issues.  This allows developers to devote more time to fulfilling high level functional requirements and bringing the system online as quickly as possible, and then modifying the system when the users inevitably discover shortcomings in its design.  

The clear winner among all alternative approaches with respect to speed of implementation and modification is Entity Beans with Container Managed Persistence.  This is also clearly the worst approach with respect to performance.  Therefore, once the requirements for the system have been validated and corrected through actual usage, we may discover that CMP-based entity beans result in unacceptable performance.  If this turns out to be true, then we will revisit this layer of the architecture and redesign it using one of the other approaches in order to bring its performance up to suitable levels.  Alternatively, we might discover that performance is adequate with CMP Entity Beans, or we might discover that the performance bottleneck is actually someplace else in the system.   This is more likely than it might seem, because the session beans can and should create caches of frequently used information.  For example, the course, requisite, and section information should probably be retrieved from the EIS Tier and cached in its entirety in the CourseMgr session instance in the EJB Tier upon system startup.  This will eliminate almost all further access to the EIS Tier by the CourseMgr.  Under these circumstances,  there will be virtually no performance cost associated with CMP Entity beans for course information, and substantial development benefits. 

CMP Entity beans have a straightforward structure: there is one entity bean per table, and each entity bean instance is associated with a row in the table. Thus, the initial CMP Entity Bean architecture will directly reflect the underlying EIS relational table structure, which is presented in the next section.

Package-level structure

The final architectural consideration is the organization of the components into a java package hierarchy. This hierarchy also corresponds to the subdirectory hierarchy.  The package level structure has the following components. First, the "root" of all packages and subpackages is "edu.hawaii.Inca".  (Note that many of you are accustomed to adding your user name to packages from previous assignments, such as "xml_johnson". Do NOT add your username to these packages--you are coding for real, now!)

Beneath the root package is the "component" name, such as "control", "approval", etc.  Each component is broken into one or more "subcomponents", such as "web", "ejb", "model", etc. Inside the subcomponents are the actual classes in the system, such as ApprovalTableModel.java, etc.

Inca Software Requirements Specification

Usage scenarios

This section presents a set of usage scenarios that provide a sense for the way the system interacts with students and administrators over the course of a semester.  The following scenarios include screen images along with links to the actual page shown.  These images have no graphical elegance, and are intended to provide a simple illustration of the functionality of the site.    Note that in several cases, the contents of the page cannot be displayed in a single screen image.  In this case, you can follow the link below the screen image to the page where the full contents appear.

Initial configuration by the Inca administrator

It's October 15, and the ICS Inca administrator, Lei Wakayama, is ready to set up the system for the upcoming semester.  After checking with the ICS sysadmin to see that the system is up and running, she goes to the Inca home page at http://Inca.ics.hawaii.edu/admin and logs in:

AdminLogin.jsp

The system is installed with a distinguished ID and password (which can be changed) for the Inca administrator(s). After Lei logs in with her ID and password, the following screen appears:

AdminMain.jsp

Note that because this is the first time Lei has logged into this system, and because no data for this semester has been uploaded, the site status is relatively uninteresting.

To configure the site for an upcoming registration season, Lei needs to accomplish the following:

Upload a file containing data on which courses are to be offered during the upcoming semester.
The data includes: course number, course name, semester and year that course is offered, sections, times they meet, prerequisites, corequisites, credit hours, and the total number of seats in each section.  Lei clicks on the "Upload semester data" link to go to a page where she specifies that file and uploads it into the system.  (Note that a separate utility for defining these files will be created.  This will allow us to define the course schedule for a semester in one file and use it for multiple purposes in the department.)
  

Upload a file containing course prerequisite data.  
This file specifies the prerequisite or corequisite courses for a given course, as well as the minimum grade required for the pre-requisite or corequisite.  It is also possible to specify a concurrent course.  Finally, instructor consent can be specified as the prerequisite (such as for an independent study course). 
 
Upload a file containing data on students prior course history and declared major.
This data includes: student name, ID, level (grad or undergrad), program number, program name (i.e. major), and a list of the ICS courses they have taken, the semester they took them in, and the grade they received (which can be empty if the course is currently in progress.) Lei clicks on the "Upload student data" link to go to a page where she specifies a file and uploads it into the system. (Note that this file is generated from ISIS data supplied to the department each semester.)  Also note that, in general, information on students is accumulated over the semesters and is not regenerated from scratch at the start of each semester.  
  
Upload a file containing approval code data.
The registration office supplies a file containing approval codes for each course, which must be loaded into the system. 
 
Specify the set of graduating senior ICS students.
Lei begins this process by using her personal records to determine which students are graduating seniors, and how many additional ICS courses they'll need to graduate this semester. She then uses  the "Edit student data" link to add this information to the Inca database. 

At this point, Inca is ready to be opened to students.  First, however, Lei decides to check and make sure that the data she uploaded appears to be correct.  To do this, she generates two reports: one that shows the set of courses defined for the Spring, 2000 semester using the "Courses" link, and one that shows the set of defined students in the system using the "Students" link. (Note that a student can become defined within the system implicitly via an ISIS record, or explicitly via their self-registration.)   

After sanity checking these reports, Lei uses the "Update news" link to tell students on the front page when the site will close for approval processing in November and when they can expect to receive the initial approvals. 

Finally, Lei uses the "Toggle student logins" link to enable student access to the website, and sends a broadcast email to uhm-ics-students@hawaii.edu.  In this email, Lei tells students to check the ICS department website for information about Spring, 2000 courses, then login to Inca to make their requests.

Melissa now goes back to the main page and selects "Create/modify course requests".  The following page appears:

CourseRequestQualified.jsp

This page is the heart of the Inca system.  First, note that the screen image above only shows a portion of the total page; in reality, Melissa is also qualified for several other courses available during the Spring 2000 semester (ICS 413, 464, and 491) which are displayed below the entry for ICS 331.

Second, the information displayed with each course combines together useful information from a variety of sources.  The "Course Qualifications" information is normally available from the Course Catalog.  The "Sections", "Corequisite Sections", and "Total seats" information is normally available from Pa'e.  The "Your situation", "Current requests", and "Current approvals" information is not usually available to students at all, but provides very useful information to students on the state of registration and their chances of getting into a section. 

The "Your situation" row provides three crucial data items.  First, it specifies whether the student is qualified for the course in question.  The default mode for this page is to display only those courses for which the student is qualified, so in the current example, this field will always be "yes".  In a later scenario, you will see a situation where the student displays all courses for the Spring semester. In this case, when a student is not qualified for a course, this field describes how the student failed to qualify for the course. 

The second data item in the "Your situation" row specifies the fine-grained priority value for Melissa Chow in ICS 331. In this example, it is 5.  This value was computed using the fine-grained priority rule examples in Chapter 4, and the only rule that applied related to Melissa's GPA.  This value is a link to a page that essentially runs the rules in verbose mode, generating an explanation for how the value was computed.  

The third data item is perhaps the most controversial: it specifies, based upon the fine-grained priority (and in the case of equal priorities, time of request submission), the ranking of Melissa's request at this moment if she were to request this course.  In this example, Melissa's request would currently be ranked 15th out of the 23 requests received so far.  Since there are 45 seats to be allocated, this appears to give her a pretty good chance at this course.  However, if 30 students with higher fine-grained priorities request this course between now and the time approvals are allocated, she would fall to 46th in line and not receive an approval code, at least until the start of the semester. Thus, the problem with providing this information is that it might give some students a false sense of their chances for a course if they submit their requests early.  On the other hand, providing this information gives those students who actively monitor their status almost real-time information about their likely chances to obtain an approval code.  This has several expected benefits.  First, it reduces student surprise when they eventually receive the email informing them that their request has been denied. Second, it allows students who discover that they have little chance of obtaining an approval code for a course the chance to look for another course in which they have better chances.  This can improve overall student satisfaction.  Third, the use of fine-grained priority rules directly ties a student's chances for course approval codes to things like their ICS GPA, their lack of having withdrawn or failed prior ICS courses, their admittance into the Honors program, and so forth.  Creating a direct relationship between these factors and their course selection priority creates an additional incentive for students to excel in our program. 

Melissa scrolls through this page and specifies her first choice as ICS 491-1, second choice as ICS 464, and third choice as ICS 413.  She presses the "Submit" button at the bottom of the page and the following page is returned as a result:

Manual registration (new/transferring students)

Lily Bender is a transferring student from the University of Washington who has never enrolled in a course at UH before, but who has been admitted as a transfer student and wants to register for courses starting in Spring 2000.  In early November, she is surfing the ICS website and learns about the Inca system. She goes to the home (login) page and learns she needs to register first.  After she visits the registration page, she learns that she is a new student and needs to contact uhmics@hawaii.edu in order to be manually registered in Inca.  So, Lily sends an email to uhmics explaining her situation.

Editing Student Data

It will occasionally happen that student records need to be manually updated by the Inca administrator. For example, a student might be accepted into the Honors Program and want to have that added to their records so that their priority values for courses increase.  Or, as in the case of our transfer student named Lily Bender, she has taken the equivalent of ICS 111 at the University of Washington and wants to have that information recorded so that she is qualified to request an approval code for ICS 211.  

To edit student data, Lei goes to the Administrator Main Page and clicks on the "Edit Current Student" link.  This results in the following page appearing:

RetrieveStudent.jsp

If she knows the student's ID, she can enter it to retrieve that student's records.  If not, she can click on the other button to generate a list of all currently defined students (first and last name and student ID).  By each student's name is a button that can be used to retrieve that student's data for editing.  In this scenario, Lei wants to edit Lily Bender's data, and she knows her student ID because she just entered it, so she types in "456234123", presses "Retrieve ID data", and the following page appears:

EditStudent.jsp

This page contains four sections: basic data, course history, requests and approvals, and consents. It lists data concerning that student for each category of information, along with a button that allows the administrator to edit information pertaining to that section. Lei clicks on "Edit ICS Course History Information" which brings up the following page:

EditStudentCourseHistory.jsp

 

This page allows Lei to enter up to seven new courses for Lily Bender.  Note that had Lily been a continuing student, the courses she had already taken would have been displayed along with the option to delete them.  Lei changes the first row to indicate that Lily has taken ICS 111 and received a grade of B in Fall, 1998.  In the comment field, Lei enters "UWash" so that future administrators will know where this course came from. 

In order to modify existing course info using this form, Lei must "delete" the existing course and simultaneously define a new one with the modified information. 

Lily can now login to Inca from Washington.  Since her record will show ICS 111 as having been taken with a B, she will be qualified to request ICS 141, ICS 211, and ICS 464.   Since she is a new student, her request is automatically placed at the end of the line, behind all continuing students, and it will be early January before she learns whether or not she will receive an approval code. Note that the situation is not quite as dire as it might appear, since only 50% of the seats in ICS 211 will be allocated before January.

Request revision

A few weeks later, Melissa Chow, our ICS major, realizes that she only really wants to take two ICS courses,  and that the course she wants to drop is ICS 491-1, which was her first choice.  Since this was her first choice, it is to her advantage to re-request ICS 464 and ICS 413 as her first and second choice, rather than maintaining them as her second and third choice. She accomplishes this in the following way:

  1. She logs in to Inca
  2. She goes to her status page
  3. She presses "Delete" for all three requests
  4. She goes to the Request page, and specifies ICS 464 as her first choice, and ICS 413 as her second choice.  
  5. She presses submit.  The system checks her requests for validity, and successfully processes her request.
  6. She goes to her student status page to verify her new set of requests.

Student review of unqualified courses

By default, the Course Request screen displays only those courses for which the student is qualified.  Course Qualification involves the following:
For each course, extracting the prerequisites. For qualification, relevant prerequisites are courses that must have been taken previously and the minimum grade in those courses, or instructor consent.
Matching the prerequisites against the student's course history and current set of consents. 

From the student main page and the course request page, students can also generate a list of all courses offered, both those that they are qualified for and those that they are not qualified for.  This page is similar to the course request page, except that the student cannot make a request from the page, and the "Qualified to request approval" field might contain an alternative to "Yes".   The alternatives are as follows:
No (Missing prerequisite course: <course number>)
No (Missing prerequisite course grade: <course number>, <grade>)
No (Instructor consent required, email instructor for further information)
No (Course already completed with B or better)

New approval allocation

On November 21, Lei logs back in to the Inca system as administrator to begin the approval allocation process. 

She begins at the Administrator main page where she clicks on the "Toggle student logins" link, which disables all further access to site by students who have not yet logged in. Those who are currently logged in can finish their activities.  

Next, she clicks on the "Edit News" link to replace the Inca home (login) page news message with one that says that the Inca system is closed to students for the next several hours for initial approval allocation. 

Next, she looks at the Site Status table, and sees that two students are currently logged in. She waits 30 minutes, then presses the "Refresh" button to refresh the site status data, and finds that the students are now logged out.  Now she can begin looking at the set of requests without worrying about whether or not additional students will make requests. 

Once the site is closed to students, Lei begins by the new approval allocation process by clicking on the "Allocate new approvals" link from the Administrator Main Page. This brings up an initial page called "Set Allocation Policy" that includes a list of all the courses.  For each course, the pages lists the number of total seats in the course across all sections, the total number of requests, the total number of approvals allocated thus far, the number of seats that would remain if all approvals resulted in a registration events, and  an "allocation"  field in which Lei can enter an integer indicating the number of new approval codes to allocate for each course.  This can be any non-negative integer.   

For example, if there are 20 seats in the course, and 10 current requests, then an allocation of 5 indicates that the five top-ranked requests should be emailed an approval code. An allocation of 0 indicates that none should get an approval code at this time.  An allocation of 15 indicates that all 10 requests should be satisfied (it is legal but uninteresting to have an allocation higher than the number of requests, which results in one approval for all current requests).   Here is a section of that page:

SetAllocation.jsp

Lei specifies "New Approvals" for around half of the seats in the intro courses, and three quarters of the seats in the 300 and 400 level courses. 

Next, Lei clicks on the "Allocate Approvals To Students" button at the bottom of the Set Allocation Policy page.  This takes her to a new page organized into two parts.  The first part is organized by course, and lists a course name followed by all of the requests for that course.  The requests are ordered by priority, and  include the student name, priority value, and a checkbox. The checkbox is checked by default if the student's request will be given an approval based upon their priority value and the number of approvals to be allocated.  It is unchecked by default if the request will not get an approval value based upon the request's priority value and the number of approvals to be allocated.   The following page illustrates the first part of the page:

AllocateNewApprovals.jsp

This page's contents results from the administrator having set the approval allocation at 5 for ICS 313, and there being 12 outstanding requests for this course.  In this case, the top five requests are checked, indicating that if the allocation was submitted without any changes, these students would be emailed an approval code for this course. Each entry includes a checkmark, a bracketed number indicating the priority value obtained by this student for this course, the student's name, and the student's ID. 

In order for Lei to make sure that this is a reasonable set of approvals, she may want to look at the current state of requests and approvals organized by student.  The second part of the page contains this report. It  lists each student who has submitted one or more requests and the courses they have requested.  If, based upon the automatic allocation, the student is going to receive an approval for the course requested, the course is listed in green.  If not, the course is listed in red. If the student has already received an approval for a course, it is listed in black. 

 Lei uses this list to attempt to ensure that, for example, every student will get at least one ICS course.  If Lei discovers students who really need a course but have not gotten it (or have gotten a course they shouldn't), then she can go back to the first part of the report and check (or uncheck) the box associated with that particular request.  Manually checking or unchecking the box overrules the default allocation algorithm.  In this case, she might review Stuart Etheridge's situation, since he is not getting any courses at all so far. 

When Lei is satisfied that the allocations look good, she can press "Submit" at the bottom of the page.  This generates emails to every student with a course request, indicating its current status.  Those who got an approval will have the approval code listed next to the request. 

Having finished with initial allocations, Lei reopens the site for further course requests by students.

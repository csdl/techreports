Welcome to the installation guide for Hackystat.  Hackystat is designed to explore the mechanisms required for (and implications of) fully automated software developer data collection and analysis.  For more information on the goals of Hackystat, see the Hackystat Research Page. 

0. Do you need to read these instructions?

For most Hackystat users, the answer is no.  If you simply want to install Hackystat sensors in order to collect and analyze your software engineering data, and if there is a Hackystat server available for your use, then what you should do instead of reading these instructions is:

  1. Open your browser and go to the home page of the Hackystat server. (CSDL maintains a public hackystat server; contact Philip Johnson for the URL.)

  2. Follow the links on the server home page to the installation instructions for the sensors you wish to install.

  3. Install the sensors.

  4. Restart your tools, and Hackystat will begin collecting and analyzing your data. 

If, on the other hand, you are interested in installing your own Hackystat server, and/or doing development on the Hackystat code base, then read on. 

1. Hackystat Distributions

Hackystat is distributed in three flavors:

  1. A "binary" distribution.  This distribution provides just enough files to get the server up and running.  Once you have the server running, you can download and install the sensor code from the server. Binary distributions are distributed as .zip files, such as hackystat-pota-bin.zip.  The top-level directory will be named with a label indicating the release name, such as "hackystat-pota" (for the Planet of the Apes release). 
      

  2. A "source snapshot" distribution.  This distribution provides the binary distribution plus all of the source files and allows you to recompile and rebuild. Source snapshot distributions are also distributed as .zip files, such as hackystat-pota-src.zip, and the top-level directory includes the release label.  Normally, the only reason to download the source distribution is to evaluate the system and decide whether you want to become a developer.  If so, you'll want to discard the source distribution and download the developer distribution. 
      

  3. A "developer" distribution.  This distribution enables you to participate in the actual development of the system. You need to have CVS installed and obtain the CVS password to the Hackystat repository. The top-level directory for CVS distributions is called "hackystat" regardless of the release currently being worked on.

The hackystat distribution is structured in terms of a number of subdirectories.  The top-level directory in the table below uses the CVS distribution name (i.e. hackystat) rather than the source or binary distribution name that includes the release label (ie. hackystat-pota). Not all directories are available in all distributions; the table below indicates their availability.
  
Directory/File Available in Contents
hackystat/ dev, src, bin Contains this document and various "build" files used to compile and package the system.
hackystat/LICENSE.txt dev, src, bin Hackystat is distributed under the terms of the GNU public license.
hackystat/build/ dev, src Subdirectory that is created by the Ant build process containing the executables and jar file. (Not present until you run the build script for the first time.)
hackystat/api dev, src Subdirectory containing the JavaDoc-generated design documentation for the system. 
hackystat/bin/ dev, src, bin Subdirectory containing various scripts to invoke the hackystat tools.
hackystat/descriptors/ dev, src, bin Subdirectory containing the XML descriptors for each notification type that are deployed onto the server at server installation time.
hackystat/webapps/ dev, src, bin Subdirectory containing various files and subdirectories required by SOAP, Cocoon, and Hackystat.  They will be copied into the Tomcat webapps directory as part of the installation process.
hackystat/lib/ dev, src, bin Subdirectory containing various jar files for development and use of Hackystat.
hackystat/src/ dev, src Subdirectory containing the sources for the system.

3. Installing the distributions

This build system is based on Jakarta Ant.  Ant uses a build file written in XML (build.xml) as building instructions. For more information refer to http://jakarta.apache.org/ant/.

Hackystat installation involves the following general steps:

  1. Download and install the infrastructure technologies (java, tomcat)
  2. Configure tomcat for hackystat..
  3. Bring up the hackystat server.
  4. Download and install the sensors for tools from the hackystat server.

The following directions require several environment variables to be set.  Procedures for setting environment variables differ widely across Windows and Unix platforms, and these instructions assume you know how to set environment variables on the platform you are using. The screenshots are generated from an installation on Windows2000, but Hackystat has also been successfully installed on other Windows and Unix platforms including Windows98, Windows/NT, Solaris 7, and RedHat Linux.

A. Download and install Java

  • Download Java from http://java.sun.com/.  The version should be 1.3 or above. 
  • Set the environment variable JAVA_HOME to point to the Java installation directory.
  • Test your Java installation by invoking JAVA_HOME/bin/java.exe -version.  It should print out version information 

B. Download and install Tomcat

  • Download Tomcat from http://jakarta.apache.org/tomcat/.  Please download version 3.2.x.  Version 4.0 is a complete rewrite and we do not expect that these instructions will work on Tomcat V4.0.  
  • Set the environment variable TOMCAT_HOME to point to the Tomcat installation directory.
  • Read the Tomcat installation instructions included with the distribution.
  • Test your Tomcat installation by invoking the TOMCAT_HOME\bin\startup script and retrieving the Tomcat home page. The following three screen shots show: (1) the invocation of the startup script, (2) the tomcat startup output, and (3) the Tomcat home page displayed in the web browser after browsing to http://localhost:8080/. 

Note that the current hackystat installation process assumes that this Tomcat installation will be dedicated to Hackystat.  If you are using Tomcat elsewhere in your environment, it is safest to download a fresh copy that is used just for Hackystat 

  • Now bring down your Tomcat server using the TOMCAT_HOME\bin\shutdown script:

C. Download Hackystat

  • Download a hackystat distribution (either source, binary, or developer).  Contact Philip Johnson for details on how to obtain the distribution.  
  • Note that due to a bug in java.util.zip, the directory permissions are munged if you unzip in Unix, and you won't be able to cd into the hackystat directory.  To fix this, just invoke the following:
    chmod -R a+x <hackystat distribution directory>

D. Configure Hackystat environment variables

  • Set the following hackystat environment variables, which are used by the Ant build scripts
    Env variable (upper case required) Example value (set to values appropriate for your installation)
    HACKYSTAT_HOST http://localhost:8080/  (note the trailing slash is required)
    HACKYSTAT_HOME c:\hackystat  (note that NO trailing slash is required)
    HACKYSTAT_MAIL_SERVER smtp-server.hawaii.edu
    HACKYSTAT_ADMIN_EMAIL johnson@hawaii.edu

  • Set the TOMCAT_OPTS environment variable to provide the Tomcat server with access to the above values through the Java system properties object.  This involves the use of the "-D" option.  Make sure the property keys are lowercase as noted below, and substitute the example property values below with values appropriate to your installation as above:
    TOMCAT_OPTS -Dhackystat_host=http://localhost:8080/
    -Dhackystat_mail_server=smtp-server.hawaii.edu
    -Dhackystat_admin_email=johnson@hawaii.edu

    Note that these "-D" arguments should all be on one line although they may not appear that way in these installation notes.
      

  • There are two other environment variables used by Hackystat that you do not need to set for a basic installation of the server.  The HACKYSTAT_USER_EMAIL environment email is used by some sensors to determine the user that data is being collected for.  The JBUILDER_HOME environment variable is needed to recompile the JBuilder sensor code. 

 

E. Build the hackystat.jar executable

Whether or not you do this step depends upon the distribution you obtained.  The binary distribution includes the hackystat.jar file in the lib directory, so you do not need to build hackystat.jar.  Moreover, since it does not include source, you cannot build hackystat.jar.  The source distribution includes the hackystat.jar file in the lib directory, and does include source, so this step is optional and you can skip it if you like.  In fact, skipping it is strongly recommended unless you are able to recompile all of the sensor code.   The developer distribution does not  include hackystat.jar and so this step is required.

  • Now you will provide Tomcat with the appropriate library files.  To do this, invoke build configureTomcat:


 

This accomplishes the following:
  • The TOMCAT_HOME\lib directory gets the executables required by SOAP, Cocoon, and Hackystat:
    • SOAP: soap.jar, apache_xerces.jar, mail.jar, activation.jar, xml4j.jar
    • Cocoon: cocoon.jar, fop_0_15_0.jar, apache_xerces.jar, turbine_pool.jar, w3c.jar, xalan_1_2_D02.jar 
    • Hackystat: hackystat.jar, jdom.jar, sax2.jar 
  • The TOMCAT_HOME\webapps directory gets the following subdirectories and associated files:
    • webapps/soap
    • webapps/cocoon
    • webapps/hackystat

G. Remove jaxp.jar

  • Remove the file TOMCAT_HOME\lib\jaxp.jar if present. (apache_xerces will be used for XML parsing.)  Note that this step may not be necessary, but has been reported to avoid problems in Cocoon.

H. Edit the Tomcat server.xml file

  • Add the following context entries to TOMCAT_HOME\conf\server.xml.  (Don't cut and paste from this document; it appears to generate parser errors.)
      <Context path="/apache-soap"
           docBase="webapps/soap"
           debug="1"
           reloadable="false">
  </Context>

  <Context path="/cocoon"
           docBase="webapps/cocoon"
           debug="0"
           reloadable="false" >
  </Context>

  <Context path="/hackystat"
           docBase="webapps/hackystat"
           debug="0"
           reloadable="false" >
  </Context>
  • Edit the following RequestInterceptor entry in TOMCAT_HOME\conf\server.xml, setting suppress="true" so that directory listings are not shown. This is critical so that users cannot browse into other user's data areas.  Here's what it should look like after modification:
    <RequestInterceptor
  className="org.apache.tomcat.request.StaticInterceptor"
  debug="0" suppress="true" />
  • By default, Tomcat listens for http requests on port 8080. If you want to run Hackystat on some other port, you need to edit the following entry in server.xml to change the port from 8080 to your preferred port number.
    <Connector className="org.apache.tomcat.service.PoolTcpConnector">
  <Parameter name="handler" 
             value="org.apache.tomcat.service.http.HttpConnectionHandler"/>
  <Parameter name="port" 
             value="8080"/>
 </Connector>
    Remember to also change the port value in HACKYSTAT_HOST and TOMCAT_OPTS!

I. Test the configured Tomcat installation.

To test your Tomcat configuration,  invoke the TOMCAT_HOME\bin\startup script again, and check to make sure that no message appears concerning the use of a "static" classpath. If so, you need to edit the tomcat.bat file to set the CLASSPATH correctly. Otherwise, you should see all of the files you copied into the lib directory in the tomcat classpath.  Here's what the command prompt window and the tomcat output should look like after configuration:

If everything is OK so far, then tomcat should now be running. Try retrieving the following pages to check the configuration and make sure that Tomcat, Soap and Cocoon are available (substituting your own host address for localhost:8080 if necessary):

  • Tomcat home page: http://localhost:8080/

  • Soap admin page: http://localhost:8080/apache-soap/admin

  • Cocoon admin page:  http://localhost:8080/cocoon/Cocoon.xml  
    (Note that there is no actual file named Cocoon.xml in the distribution--this URL invokes a Cocoon servlet.)

K. Run the JUnit tests

To finish verifying your installation, you should run the Hackystat JUnit test suite.  Invoke: build testInstall.

Do not be misled by the "BUILD SUCCESSFUL" notification: that does not mean the tests all passed successfully.  You need to make sure that "Failures: 0" appears for each test case as in the example screen dump above.  

If all the tests passed successfully, congratulations!  You have successfully installed the Hackystat server.

M.  Play with the sample application SimpleTimer (Optional)

SimpleTimer is a simple application designed to show how data can be sent to a Hackystat server.  Unlike real sensors, which are downloaded from the server and which send data automatically, SimpleTimer is an application that is included with the server distribution only and which requires the user to manually send time data.  Try the following once you have the hackystat server up and running:

  • Make sure the following environment variables are set:
    Env variable (upper case required) Example value (set to values appropriate for your installation)
    HACKYSTAT_HOST http://localhost:8080/  (note the trailing slash is required)
    HACKYSTAT_HOME u:\cvs\hackystat
    HACKYSTAT_USER_EMAIL johnson@hawaii.edu
    JAVA_HOME c:\jdk1.3

     

  • Invoke SimpleTimer.  From the hackystat directory, in windows, "bin\runSimpleTimer" will work.  There is a runSimpleTimer.sh script if you're using Unix. Here's the invocation and the resulting SimpleTimer window showing the timer collecting some time:

  • To collect some time, press the start button icon (a triangle).  Once time is being collected, the start button icon turns into a stop button icon (a square).  When you're ready to stop collecting time, press the stop button icon.  To send the data off to the Hackystat server, press the "Record" button. There is also a pause button icon (two vertical bars) that toggle pausing of time data collection. Finally, the text field allows you to enter the names of the files you were working on.  SimpleTimer is not intended to be used for real data collection, but rather to illustrate to developers and initial users how data is collected and sent. 
      
    If this is the first time you've sent data to this hackystat server, you'll subsequently receive an email message from the server providing you with links to your personal data area.

6. FAQ

  1. I get the following error when I try to "build deploy": 
    
[SOAPException: faultCode=SOAP-ENV:Protocol;
msg=Unsupported response content type "text/html;
charset=ISO8859-1", must be: "text/xml".

(more stuff follows)
    This message typically occurs when the server can't be found, and most likely indicates that your HACKYSTAT_HOST value is incorrect. Make sure, for example, that it includes the "8080" port number if you're running on the default Tomcat port.
      
  2. I am using Windows 95/98/ME and get the following error: 
    
Out of environment space
    This is an infamous Windows "bug". To fix it on Windows 95/98, you need to edit your config.sys file and add the following line: 
    shell=C:\command.com /e:4096 /p
    (Or whatever the path is to command.com on your system.) 
      
    On WindowsME, config.sys statements are ignored and you have to include your environment variables in the Autoexec.bat file. See the Microsoft Knowledge Base for more details. To ensure that the changes take effect, you should restart your computer.  
     
  3. I am using Windows 95/98/ME, and Step 4 fails when trying to retrieve the Cocoon.xml admin servlet.
      
    This is probably due to the failure of Tomcat to automatically set up its classpath correctly.  You will have to set the classpath manually in either the tomcat.bat file or in your autoexec.bat file.  The classpath used by tomcat should include all of the jar files in the tomcat/lib directory (after "build install") in alphabetical order.
     
  4. I get the following error message when I try to retrieve Cocoon.xml in Step 4:
    
  Publishing Engine could not be initialized.
    This is probably due to apache_xerces.jar not being first in the tomcat classpath, or parser.jar coming before apache_xerces.jar in the Tomcat classpath.  One potential way to fix this is to change the line 
    SET CP=%CP%;%CLASSPATH%
    into 
    SET CP=%CLASSPATH%;%CP%
    Other approaches can work as long as they result in the classpath including all the .jar files in tomcat/lib and their being listed in alphabetical order. This happens by default when Tomcat succeeds in building the classpath automatically.

1.0 Introduction

Aloha. I am Michael Paulding, a second year Ph.D. student at the University of Hawai'i - Manoa (UHM).  My research advisor is Dr. Philip Johnson and we work in the Collaborative Software Development Laboratory (CSDL) at UHM.  My current research interests include the analysis of processes and products of software engineering inherit to the HPCS community.  In the spring of 2004, I began work on an implementation of the Optimal Truss Purpose-Based Benchmark (PBB), work which was originally initiated by John Gustafson at Sun Microsystems.  During the summer of 2004, I spent six weeks at Sun Labs in Mountain View, CA developing automatic tool support for HPCS size metrics in conjunction with further work on the Truss PBB.

Recently, I have been analyzing data collected during the PBB development process, including process and product data captured automatically by CSDL's Hackystat system and data recorded by hand.  In particular, I have been trying to characterize my work under the context of productivity to isolate and understand workflow trends and productivity bottlenecks and enhancers.  This technical report provides an overview of my progress so far. 

2.0 Problem Overview

To contribute to the community of purpose-based benchmarks, I have been working on an implementation of the Optimal Truss PBB intermittently over the past several months.  This project was undertaken with the hope of achieving two specific goals, namely:

  1. Use the theory and algorithms of parallel computing learned in a classroom setting and apply them to a real, tangible HPCS problem (get down in the trenches).
  2. Attempt to surface and understand the software engineering issues inherit to HPCS development.

The Optimal Truss PBB problem can be summarized by the following description:

Given a set of 3 attachment points on a vertical surface and 1 point away from that surface that must bear a load:

Find a pin-connected steel truss structure that uses as little steel as possible to bear a load with a 50% safety factor.

4.0 HPCS Workflow Trends Uncovered by Hackystat

One of the analyses that can be performed on Hackystat data is a workflow characterization.  By reviewing command line invocations from the console, it is often possible to gain insight into what I was doing on a particular day with respect to workflow.

Before I delve into the analysis of workflow, some background information is needed about my design philosophy with regards to the Optimal Truss PBB.  In the early stages of development, I focused on creating a "sequential workhorse" to represent a unit of computation that required enough processor time to justify the overhead incurred by communication.  More simply stated, I wanted to make sure that the computation of this unit took more time than that required to send it from one processor to another (for task division).  On non-Hero architectures such as the one I had access to, the latency of one communication easily overwhelmed 1,000 multiplication operations.

On the other hand, I did not want the unit of computation to take significantly longer than communication time, as then I would not be maximizing the level of parallelism in the solution.

For the Optimal Truss PBB, I defined the unit of computation as the solving of a truss.  Specifically, given a set of attachment points, pin-connected members, and a load bearing point (with load), the solving of a truss involved force and moment equilibrium (Fx, Fy, Fz and Moments), deformation checks (does a member's length exceed the maximum for its material) and summation of mass.  A valid solution is one in which 1) the truss is in equilibrium (not accelerating), 2) all members meet deformation constraints, 3) total mass is a real number.

Once this sequential workhorse had been defined, the next step was to determine how the more complicated task of figuring out how to generate all possible trusses with minimum (or optimally zero) redundancy and distribute them equally among the available processors.

It would be interesting to use the Hackystat to identify high level analyses, such as the one above.  Currently, Hackystat provides the architecture for automatically collecting the raw data needed for the analysis.  With additional effort, one can use this data to understand their own HPCS development process.  For instance, a review of the daily diary assists the reader with categorizing how much time was spent in developing, debugging, testing and verification/validation.  All this information is present in command line data, which can enable one to combine the event sequence with active time to produce a metric indicating how long a developer spends in phases of HPCS development.  The following section provides a simple example of understanding workflow from Hackystat data.

4.1 Simple Workflow Analysis

The following is a simple example in which the sequence of activities reported by Hackystat indicates a workflow trend.  From the data, I will make the claim that my programming style tends to follow the pattern of:

  1. Make incremental change to the source code  (e.g. add functionality or fix bug)
  2. Compile code as level 1 verification
  3. If L1 verification fails, return to step 1
  4. Write new tests to accompany changes made to the source code
  5. Run test suite as level 2 verification
  6. If L2 verification fails, return to step 4
  7. Repeat steps 1-6 for the next incremental change

Below is partial output from the Hackystat diary that supports this pattern.

 


The data shows that I edited the point class file for approximately 2 minutes (step 1) during the interval between 7:45-7:50pm, indicated by the "(Edit File 120s) point.cc" statement.  Also during this interval, I am performing L1 verification (steps 2 and 3), indicated by the multiple compile statements (g++ ...).  In this snippet of output, it is evident that I am using the full compilation commands rather than running the "make" process.  Although "make" simplifies and makes L1 verification a routine task, its command line call does not provide the level of detail appropriate for data analysis (e.g. the "g++ -c point.cc" line would read "make ...", which would not indicate the files undergoing L1 testing).  I address this inadequacy in more detail in section 4.

From 7:50-7:55pm, the Hackystat diary shows that I am starting to perform L2 testing (step 4).  For example, I run testpoint.exe at 7:51 as part of L2 testing to ensure that the changes made to the source code did not break existing test cases.  Apparently, something in the point.h file caused the existing test suite to fail.  Therefore, at 7:52, I edit the point.h file to fix the error and verify it in the call to testpoint.exe at 7:53pm.

From 7:55-8:00pm, it is evident that I am adding test cases to support the incremental change in the source code by the "(Edit File 60s) testpoint.cc" statement at 7:56pm.  To complete L2 verification, I run the testpoint.exe suite to ensure that the code passes (steps 5 and 6).  It does, and I begin the next incremental change to the source code, restarting at step 1.

 

5.0 Productivity Bottlenecks Uncovered by Developer Journal Analysis

Hackystat data is quite useful for understanding the "process" aspects of my PBB development, but it does not tell the whole story in regards to productivity aspects.  For instance, in the development streams of section 3.1, Hackystat does not report why I am making incremental changes to the source code.  Am I adding functionality? Or fixing a bug?  To answer these questions, we must currently use some more intrusive technique, such as interviewing the developer. In future, with additional instrumentation support, Hackystat might be able to provide better automated insight into characterizing what kind of development is occurring at various points in time. 

For now, however, my way of dealing with the limitations of Hackystat data collection and analysis has been to keep a personal journal to record information that I thought might be important to the development process, but not covered by Hackystat.  In the journals I've been maintaining during the Truss PBB development, I recorded information about defects I encountered (primarily related to HPCS), insights discovered and calibration data, such as self-reported active time and files edited.

An excerpt of this data is presented below to indicate a productivity bottleneck I encountered during development, but was not uncovered by Hackystat:

Insights:

The sequential implementation of the truss optimization problem is now reaching the point in which a linear algebra solver must be incorporated into the code to solve the system of equations imposed by the forces and moments about each joint.  Most of the available packages bear a significant memory burden as a result of the large number of libraries included.  In addition many of the linear algebra solvers are written in Fortran, therefore a proxy is needed to convert the C++ calls into the native Fortran routines provided by the package.

Another option is to build a custom linear algebra solver, but this involves a great deal of overhead in developer time, as the matrix data structures and vector operations involved in Ax = b will be written from scratch.

The entire journal entry can be found here.  A listing of all journals appears at http://csdl.ics.hawaii.edu/Research/HPC/#Logs.

As the journal entry indicates, a productivity bottleneck occurred when I was faced with the problem of solving equilibrium constraints on the truss.  The shortest path to a solution was to install third-party software (namely LAPACK) to provide the routines needed.  However, this incurred quite a bit of overhead and productivity degradation, as each machine in the cluster needed to be installed with this ~40MB package.  In addition to the memory requirements, root access to the cluster needed to be gained as well as 1-2 hours of administration time to perform and verify the install.  Since the machine was not owned by my research lab, the complete process from requesting LAPACK, to having it approved, installed and verified took 3 days.  Perhaps this time constraint could be reduced in an industry setting, but it is still a great bottleneck in productivity.

In addition, this third-party software requirement renders my source code difficult to port to other architectures.  For my implementation to execute properly, I would have to repeat the installation request on the new machine, therefore increasing the overhead by another order of magnitude.

Issues such as these are crucial to record in a developer journal for future reflection and analysis.  While a formal, stylized journal entry has its benefits, I believe it might also be sufficient to use email for this purpose, as long as the message is archived and easily retrievable.

6.0 Implications for 10x Productivity Improvement

The current infrastructure for HPCS metrics collection seems incomplete for gathering the data necessary to understand major productivity bottlenecks.  There are many tools for analyzing the performance of a particular benchmark, but I have not found tools for analyzing the development process.  Data such as active time, workflow trends and time allocation are usually self-reported in a conversation, meeting or email dialog, but not in an consistent, ordered and automatic manner.

I believe the Hackstat system in its current state can gather critical metrics such as active time, developer workflow and time allocation with little overhead.  I also believe that Hackystat can be extended to provide finer-grain analyses to isolate productivity bottlenecks.  For example, I have heard several HPCS developers and researchers note that a large portion of the total development time is dedicated to integrating sequential code with MPI.  However, I have not heard any measurements quantifying this claim.  From my own experience, I believe this hypothesis to be true, but as far as I know, the data is not available yet.

One task that interests me and which I have been pursuing is to be able to statically analyze HPCS source code (written in either Fortran, C or C++) to isolate MPI constructs and library calls.  I am planning to write software to create and traverse parse trees for the purpose of analyzing whether a particular file contains MPI directives.  Upon traversal completion, I will then be able to produce a metric quantifying the MPI occurrence(s) within a file.  Once this metric has been calculated, I can then couple it with Hackystat active time data and compare it to other files within the software package.

With an analysis such as this, I hope to classify how much of active time is devoted to sequential vs. parallel aspects of the code.  I will also be able to produce a ratio for lines of code devoted to each category. Eventually, I hope to be able to understand what particular MPI directives have consumed the most of my development times.  Finally, I will be able to show data that confirms or denies the claim that integrating sequential code with MPI consumed the majority of development time on my Optimal Truss PBB implementation.

8.0 Future Directions

An outline of the future directions for Hackystat HPCS application is as follows (hopefully to be done in parallel):

  1. Begin second iteration of Truss PBB development, including a parallel implementation of equilibrium constraints, including those of deformation, force and moment.
  2. Create a shell wrapper for the "make" utility to send specific commands to Hackystat (e.g. make truss_sequential sends g++ point.o joint.o topology.o truss.o -o truss_sequential.exe to diary)
  3. Capture results of L1 verification from the compiler (e.g. compile passed or failed)
  4. Capture results of L2 verification from the test suite (e.g. unit tests passed, failed or caused error)
  5. Add current working directory to the Hackystat diary
